📖 Quick reference — codes, timing, retries

MLS — Application Response codes

CodeStatusMeaning
ABAcknowledgedReceiver AP got the message; processing in progress.
APAcceptedValidated and delivered to the buyer.
RERejectedValidation failed. Carries a structured reason code. Also reported to C5.

MLS — Status reason codes

CodeMeaning
SVSchema validation failure (XSD)
BVBusiness validation failure (Schematron)
DLDelivered to C4
DFDelivery failed (could not reach C4)
NRNot routable (no valid endpoint)

TDD — three management functions per the official Peppol UAE TDD spec

FunctionWhen
submitOriginal report from C2 (sender) and C3 (receiver) to C5.
replaceCorrected version supersedes a previous TDD.
withdrawCloses a previous TDD — used after RE to invalidate the original positive sender-side report.

UAE-specific timing

EventWindow
C2 → C5 TDD report (sender)≤ 15 minutes from invoice timestamp
C3 → C5 TDD report (receiver)≤ 15 minutes post-validation
C3 → C2 MLS = ABImmediate (acknowledgement)
C3 → C2 MLS = AP / REAsync; after validation completes
RE handling (withdraw + correct + resubmit)Before VAT-period close
⚙️ How ClearTax handles failures (retries at scale)
Failure modeWhat we do
C3 unreachable on AS4 sendExponential backoff: 1m → 5m → 30m → 4h, then dead-letter + ops-paged.
MLS not received within SLAC2 polls C3; escalate as NOT_DELIVERED if still pending past threshold.
Orphan / in-flight envelopesHourly reconciliation worker sweeps to catch silent losses.
C5 reporting failureIndependent retry queue with circuit-breaker; ops-paged on sustained failure.
Validation rejection (RE)Auto-reason-code parsing (SV/BV), surface fix suggestions, fire TDD withdraw to C5 automatically.
